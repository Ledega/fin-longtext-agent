你可以按“两阶段工程化落地”的思路来做：先用 Step 1 搭出一个稳妥的 RAG+Agent baseline，保证能过关拿到“还可以”的分数，再用 Step 2 分阶段精细优化检索、证据聚合和 B 榜无 doc_ids 场景。 

***

## Step 1：先搭一个可靠的 RAG+Agent 基线系统

目标：在只用 Qwen 系列 API 的前提下，完成题目格式要求，正确处理 A/B 榜，并在准确率和 Token 消耗上明显优于官方“长文直接塞模型”的 baseline（15% 准确率、Token 超标）。 

### 1. 文档离线预处理与结构化

- 对提供的所有 PDF / 原始文本做统一抽取，切成“段落级 chunk”，并保留 `doc_id`、页码、章节标题、条款编号等元数据，后续检索时能直接回到原文。 
- 对财报/合同这类强结构文档，至少把“表格 + 标题 + 单元格文本”转成行列结构（例如 DataFrame / JSON），便于后续按指标名检索和比对。 
- 预处理阶段可以相对“粗糙但完整”：先保证所有文本进系统，不追求极致压缩，后续 Step 2 再做精细化压缩。 

### 2. 建索引：BM25 + 向量 + 领域过滤

- 按五个领域分别建索引：insurance / regulatory / financial_contracts / financial_reports / research，用 domain 字段先做粗筛，避免跨领域干扰。
- 每个 chunk 保存两类索引：  
  - 关键词/BM25：对条款编号、指标名称、专业术语效果好；  
  - 向量索引：对长句描述、语义相似检索效果好。
- 查询时的第一步：根据题目里的 `domain` 字段和 `type`（如“财务指标对比分析”）限定候选文档集合，再在集合里做 BM25 + 向量**混合检索**。 

### 3. A 榜：有 doc_ids 的基础问答流程

A 榜给了 `doc_ids`，这是最容易先拉分的部分。 

- 对于每道题：  
  1）从题目 JSON 获取 `doc_ids`，从这些文档中检索与题干关键词（公司名、年份、指标、金额等）最相关的若干 chunk（例如每个 doc 取 Top-k）。 
  2）把若干 chunk 组成上下文，连同题干和选项一起作为 prompt 喂给 Qwen，使用一个通用 QA 模板。 
  3）在 prompt 中明确：  
     - 先在“思考区域”一步步分析，但最终只在最后一行输出选项字母（单选 1 个字母，多选若干排序好的字母，判断题 A/B）。 
- 这样你已经具备“多文档检索 + 基本证据聚合 + 格式规范输出”，在 A 榜上能比直接塞长文的 baseline 高不少。 

### 4. B 榜：无 doc_ids 的基础处理方式

Step 1 阶段先用一个简单但有效的 B 榜方案，保证能做出来而不是跳过。 

- 对于 B 榜题目：  
  1）根据 `domain` 先锁定领域子库，例如 regulatory 就只在监管法规文档集合里检索。
  2）用题干做 BM25 + 向量检索，召回若干候选 `doc_id`（比如 Top-3 or Top-5）。 
  3）在这些候选文档内再按 A 榜流程检索 chunk + 调用 Qwen。 
- 虽然这个 B 榜方案还不够“精细”，但已经避免了 baseline 那种直接全库堆上下文的做法，同时至少能对准一部分相关文档。

### 5. Qwen 调用模板与答案 post-processing

- 针对三类题型设计三个 prompt 模板，但共用一条原则：  
  - 思维链在中间，最后单独一行输出答案字母；  
  - 在系统层用正则只抽取最后一行写入 answer.csv。
- 在 post-processing 中做：  
  - 校验是否只包含合法字母（A–D）；  
  - 多选题对字母排序、去重；  
  - 若不合法可以触发一次“降温重试”或回退策略。
- 同时记录每次 API 调用的 prompt_tokens、completion_tokens，并汇总出 summary 行中的 total_tokens，以满足评测对 Token 统计的要求。 

> 到这里，Step 1 的系统已经能：覆盖全题、正确处理题型和答案格式、合理控制 Token 数量，并利用基础检索避免 baseline 那种极端“喂全文”策略，基本能拿到一个“相对可以”的分数。

## Step 2：重点优化正确率和 B 榜无 doc_ids 场景

Step 2 的目标是：在 Step 1 的系统基础上，针对“文档解析质量、检索策略、证据聚合、领域知识和 Token 成本”这些官方点名的方向做深度优化，尤其要增强 B 榜多文档盲检和复杂推理题的表现。

### 1. 文档解析质量的增强（offline 强化）

- 对财报类文档：  
  - 重点把“财务报表三大表 + 附注”解析成结构化表格，并在索引中同时保留文本和结构信息，例如：指标名、年份、数值、单位、所在表页。 
  - 这样在比对“2025 vs 2024”的营收、净利润、现金流时，可以直接在结构化层面做差值和趋势分析，再把结果和对应摘录的文本一起给 Qwen 做核验。 
- 对保险条款、合同、监管法规：  
  - 尽量保留条款编号、章节层级（第一章、第二条、第（一）款…），使后续检索可以“按条款编号 + 关键词”联合定位，避免只靠语义相似度。 
  - 可以在离线阶段生成“条款摘要，但保留条件和例外”，用于压缩场景。 

### 2. 检索策略：从“粗召回 + Top-k”进化到“层级检索 + re-rank”

- 增加候选文档 re-ranking：  
  - 先用 BM25/向量召回更宽的候选 doc_ids（例如 Top-10），然后根据 domain 特征、实体匹配（公司名、年份、金额量级）、题型（比如“财务指标对比分析”更看重含有数字和表格的文档）做 re-rank，选少量最有可能的文档。
- 在文档内部采用两层检索：  
  1）以题干为 query 做 chunk 级检索；  
  2）以每个选项为 query，再做一轮 finer-level 检索，以便获得“选项级证据片段”。
- 针对 B 榜：可以把“先判定涉及的公司/产品/年份”作为一个子任务，让 Qwen 在小上下文上先帮你猜测应检索的主体，再据此进行目标文档检索，减少误检。

### 3. 证据聚合与“选项级推理”模板

官方示例 reg_a_014 清楚展示了他们期望的“证据可追溯”风格，你可以直接把这个写成 Agent 的通用推理模式。 

- 针对多选/复杂题，设计这样的推理流水线：  
  1）先检索每个选项对应的 1–3 条证据 chunk（可能来自不同文档）；  
  2）构造 prompt，让 Qwen 对每个选项单独说明：  
     - 关键事实摘录；  
     - 适用的条文/条款编号；  
     - 判断是否正确及理由；  
  3）最后由 Qwen 或你的规则层，把被判定为“正确”的选项字母收集起来、排序后输出。 
- 这样可以显著提升：  
  - 抗干扰能力（因为每个选项都明确和证据绑定）；  
  - 对 B 榜多文档题目的鲁棒性（即便召回多个文档，也能利用局部证据逐项判断）。 

### 4. 跨题目记忆与 Token 成本优化

- 对同一个 `doc_id`：  
  - 在第一批题目中，允许较大 Token 预算来构建“文档知识摘要”，比如：  
    - 对财报：自动抽出年度关键指标表、同比变化结论；  
    - 对保险条款：抽出各项责任触发条件、赔付公式表；  
    - 对监管法规：抽出“必须股东大会审议事项”“必须特别决议事项”等清单。 
  - 后续遇到同一 doc_id 的题目，优先在这些摘要上检索，再必要时 fallback 到原文 chunk，极大减少重复读整文。
- 引入“缓存层”：  
  - 对同一题目在调参过程中多次调用 Qwen，要保证缓存生效，避免重复消耗评测 Token；  
  - 对于 B 榜检索出的候选 doc_ids 同样缓存，避免跨题目重复计算重新检索。 

### 5. B 榜无 doc_ids 题目的专门应对策略

B 榜的关键难点是：题目不直接给 doc_ids，你要先找“可能相关的那几份长文”，再做阅读和推理。这一块的优化对总分影响非常大。 
可分三层：

1. **文档级粗召回层**  
   - 使用题干做 BM25 + 向量召回，取 Top-N 文档；  
   - 按 domain 过滤，比如监管题只在监管法规集合里找；  
   - 加入实体匹配（公司名、证券代码、年份、产品名）重打分。

2. **场景理解 + 精细筛选层**  
   - 用一个轻量级 Qwen 调用，在题干小上下文下让模型判断：  
     - 涉及哪些实体（公司、产品、法规名称）；  
     - 涉及哪些具体业务类型（例如“募集资金用途变更”“对外担保”“独立董事资格”等）。 
   - 再用这些实体和关键词对召回的候选文档做二次过滤，极大减小错误文档被选入的概率。
3. **多文档证据聚合层**  
   - 对最终选定的 2–3 份文档，按“选项级检索+推理”进行联合判断（见前一节），而不是只取一个文档。
   - 对每个选项保留“来自哪份文档、哪一条条款/章节”的信息，提升推理的稳定性和可解释性。

### 6. 针对不同领域的模板和后处理

- 为五个领域分别设计 Prompt 头部和规则：  
  - 监管法规：强调“必须严格依据条文，不使用常识”，并要求回答中注明适用条文编号（即便只在思考区域）。 
  - 保险条款：要求先列出适用公式和条件，再代入题干中的金额/状态计算，最后得出比较结果。
  - 财务报表：要求列出两个年份的具体数值，再判断“增长/下降/比例变化”，避免主观词汇。 
- 后处理层针对各领域增加规则校验，例如：  
  - 对财务比对题，规则层自己算一遍简单加减乘除，看与 Qwen 给出的结论是否一致，不一致时可以触发第二轮只给数值让 Qwen 再判一次。 
  - 对监管题，检查回答中是否引用了正确的条文编号，若没有，可以触发增强检索再问一次。 
***

总体来说，可以这样理解：

- **Step 1**：用“标准 RAG + 简单 Agent”尽快跑通完整 pipeline——PDF→chunk→索引→检索→Qwen 推理→答案 post-process→answer.csv + Token 统计，保证能覆盖所有题型、支持 A/B 榜、格式完全合规，先拿到一个稳定的、明显好于官方 baseline 的分数。 
- **Step 2**：在此基础上，重点在“解析质量、检索策略、选项级证据聚合、跨题目记忆和 B 榜无 doc_ids 检索”上精细打磨，把系统从“能做题”提升为“能稳定做对难题、还能省 Token”的版本，争取在准确率和 TokenScore 两端都压榨到位。
